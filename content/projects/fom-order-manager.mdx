## 🎯 Executive Summary & Problem Space

In many developing markets, social media platforms like **Facebook** serve as the primary storefront for small-to-medium retail businesses. Businesses list products via posts, interact with customers via Facebook Messenger, and negotiate payments manually through bank screenshot uploads.

However, this chat-first workflow introduces severe operational friction:

1. **Disconnected Silos**: Customer conversations, order details, invoices, and payment statuses are spread across different messaging platforms, notebooks, or spreadsheets.
2. **Payment Fraud & Manual Verification**: Reconciling dozens of bank transfer screenshot uploads daily is slow and prone to errors or fraud.
3. **Poor Connectivity**: Delivery personnel and shop managers lose access to crucial business data in low-bandwidth or offline zones.

**FOM Platform** addresses these challenges by consolidating social storefront chat streams into a **Unified Inbox**, standardizing order entries, automating local QR-scan payment reconciliation, and running an offline-first Flutter mobile client.

## 💡 Key Engineering Accomplishments

### 1. Extensible Webhook Channel Abstraction

Instead of hardcoding Meta Messenger integrations, the codebase features a provider-agnostic messaging architecture. Custom webhooks ingest messages from multiple channels (Messenger, Telegram, and future Viber support) and normalize them before persisting:

- **Unified Inbox Handler**: Adapters map incoming raw webhooks to a standard `ChannelInboundEvent` interface.
- **Queue-Backed Processing**: Webhooks are ingested instantly and placed into a Redis/BullMQ queue to prevent drops due to burst traffic.
- **Normalized DB Schema**: Standardized message records are automatically linked to customer profiles based on verified identifiers (phone numbers, user profiles).

### 2. Offline-First Mobile Synchronization Engine

To handle spotty mobile connections, the Flutter application leverages an offline-first database approach:

- **Local DB (Drift/SQLite)**: Complete transactional data (orders, customers, delivery queues) is cached locally on the device.
- **Background Sync Queue**: Operations executed offline are serialized into a sync queue.
- **Smart Conflict Resolution**: When the client reconnects, the engine synchronizes the pending changes with the API server, applying merge strategies based on transaction timestamps.

### 3. Automated Local QR Merchant Payments

FOM automates order validation and decreases merchant transaction verification overhead by integrating with **MyanMyanPay** (supporting KBZPay, Wave, AYA Pay, etc.):

- **Dynamic QR Codes**: Generating unique, time-sensitive QR codes with embedded metadata (order ID and expected exact amount) via `MMPaySDK`.
- **Callback Signature Verification**: Securing the incoming payment webhooks by calculating HMAC signatures using the secret merchant keys.
- **State Machine Transitions**: Automatically updating the Order status to `Confirmed` and logging the payment transaction ID upon receiving valid payment callbacks.

### 4. Database-Backed Role-Based Access Control (RBAC)

Designed a secure authorization system enabling fine-grained control:

- **Multi-Tenant Hierarchy**: Users belong to shops through membership associations.
- **JW-Token Payload Inflation**: Shop memberships, user permissions, and active roles are compiled directly into JWT access tokens. This prevents redundant database queries during client requests while maintaining real-time UI customization.

## 🗃️ Database Schema & Scale

The database uses PostgreSQL managed via Prisma. Database indexes are placed on foreign keys (`shopId`, `customerId`, `orderId`) to support swift querying under large workloads.

### Core Schema Associations

- `shops` has a many-to-many relationship with `users` via the `memberships` table.
- `shops` has a one-to-many relationship with `orders`, `customers`, `products`, and `channel_connections`.
- `orders` has a one-to-many relationship with `order_items`, `deliveries`, and `payments`.
- `channel_connections` has a one-to-many relationship with `channel_threads`, which maps to `channel_messages`.

### Key Tables & Target Database Scale

| Table Name  | Description                   | Target Record Vol. |
| :---------- | :---------------------------- | :----------------- |
| `shops`     | Registered shop accounts      | ~1,000             |
| `users`     | Staff and owner accounts      | ~5,000             |
| `orders`    | Transaction and order records | ~100,000           |
| `customers` | Customer profiling data       | ~50,000            |
| `products`  | Product listings              | ~10,000            |

## 🔒 Security, Compliance, & Performance

### Security Guardrails

- **Data Encryption**: Merchant tokens and Facebook page credentials are encrypted in PostgreSQL using AES-256-GCM.
- **Payload Validation**: Inputs verified on backend endpoints using Zod schema structures to prevent SQL injections.
- **Access Control**: Active route guards evaluate roles and permissions stored securely inside JWT payloads.

- **GDPR & CCPA**: User profile deletion procedures and data portability capabilities are built into account settings.

## 🛠️ Key Technical Challenges & Solutions

### Challenge 1: Real-Time Synchronization Across Platforms

- **Problem**: Keeping Web Dashboards and Mobile applications in sync with customer messages and order updates in real-time.
- **Solution**: Developed a persistent WebSocket layer using NestJS and Socket.IO. We used Redis as a Pub/Sub adapter to sync event states across container instances. On the mobile client, changes update Flutter state blocks (`flutter_bloc`) instantly without full-screen page refreshes.

### Challenge 2: Client Synchronization in Low-Connectivity Regions

- **Problem**: Delivery drivers frequently lose network coverage, leading to data loss during deliveries.
- **Solution**: Deployed a local background queue in Flutter utilizing the `Drift` database. Every local mutation is logged as a pending task. When connectivity returns, a sync service transmits events, resolves data conflicts with timestamp merging, and updates local records.

### Challenge 3: Extensible Message Webhook Scaling

- **Problem**: High-volume chat streams during marketing campaigns could overwhelm the webhook endpoints.
- **Solution**: Decoupled webhook ingestion from parsing. Inbound webhooks are immediately queued inside Redis using `BullMQ`. Worker threads process individual queue items asynchronously, ensuring the HTTP API remains available under load.
